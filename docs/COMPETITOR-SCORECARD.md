# Competitor Scorecard

The SKS 5.9.0 competitor scorecard measures release readiness from real local artifacts. It is deliberately not a brand or documentation score.

The scorecard is generated by `dist/scripts/competitor-scorecard-check.js` after `npm run build:incremental`. The script reads `config/competitor-scorecard-baseline.json`, writes `.sneakoscope/reports/competitor-scorecard.json`, and exits non-zero when the total score is below `94` or any category is below `90`.

Each category is scored from evidence artifacts listed in the baseline config:

| Category | Evidence |
|---|---|
| `code_stability` | doctor idempotence, real evidence policy, route regression |
| `test_release_gates` | release check stamp, installed package smoke, full release check |
| `parallel_isolation` | parallel runtime proof, worktree runtime, patch merge proof |
| `speed_performance` | perf budget, hook latency, gate timing |
| `install_operations` | installed package smoke, doctor idempotence, publish dry report |
| `maintainability` | architecture check, route modularity check, command budget check, orphan scan |

Missing evidence earns zero for that evidence item. Evidence must be JSON and must look passing through explicit fields such as `ok: true`, `passed: true`, `success: true`, or `status: "passed"` without blockers, errors, or failures. Missing or blocked artifacts are reported in `blockers`.

This means early 5.9.0 work is expected to fail the scorecard until the corresponding runtime proofs and smoke artifacts exist. That failure is the point: release quality should be artifact-driven, not hardcoded.
